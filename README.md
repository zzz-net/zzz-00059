# 活动场地排期系统

本地运行的活动场地申请、审批、排期管理系统。核心围绕**申请-审批-冲突检测-历史链路**设计，不是简单的预约 CRUD。

## 功能特性

### 核心业务
- **场地管理**：维护场地信息、开放时段、日配额
- **申请流程**：提交 → 待审批 → 已确认 / 已驳回 / 已取消
- **冲突检测**：时段重叠校验、日配额校验
- **状态回退**：已取消申请可撤销回之前状态（会再次校验冲突）
- **历史链路**：每次状态变更都记录操作人、时间、原因，形成完整审计链
- **操作日志**：全局审计日志，记录所有关键操作
- **排期导出**：按日期导出 CSV 排期表

### 关键校验规则
1. **创建时**：时段必须在场地营业时间内
2. **审批时**：同场地同时段已确认的申请会拦截后者
3. **审批时**：当日已确认数量不能超过日配额
4. **撤销取消时**：重新检查时段冲突和日配额

### 权限规则
- **审批人**：可以审批通过/驳回、撤销取消、取消任意申请
- **普通申请人**：只能提交申请、取消自己的申请，不能审批
- 默认审批人名单：`张三`、`管理员`、`admin`、`Administrator`（可在 `app.py` 的 `APPROVERS` 中修改）

## 本地启动

### 环境要求
- Python 3.8+

### 步骤

```bash
# 1. 进入项目目录
cd zzz-00059

# 2. 安装依赖
pip install -r requirements.txt

# 3. 启动应用（首次启动会自动建表并初始化3个示例场地）
python app.py
```

启动后访问：http://localhost:5001

数据存储在项目根目录下的 `scheduling.db`（SQLite 文件），重启不丢失。

## 验证步骤

### 主流程验证（合法申请从提交到审批通过 → 导出）

1. **启动应用**：`python app.py`
2. **查看场地**：打开首页，确认有 3 个示例场地（多功能厅A、会议室B、活动室C）
3. **新建申请**：
   - 点击「申请管理」→「+ 新建申请」
   - 选择场地「多功能厅A」，填活动名称、申请人
   - 选明天日期，开始 10:00，结束 12:00
   - 提交，状态为「待审批」
4. **审批通过**：
   - 点击「审批面板」，找到刚提交的申请
   - 点「通过」，填写审批意见
   - 状态变为「已确认」
5. **查看详情与历史**：
   - 点「详情」，可看到完整的状态历史链路：提交 → 待审批（系统流转）→ 已确认（审批人 + 意见）
6. **查看排期**：
   - 点击「排期视图」，选择对应日期
   - 可看到该场地的已确认排期
7. **导出排期**：
   - 点「导出CSV」，下载当日排期表
   - 用 Excel 打开，确认数据正确
8. **查看操作日志**：
   - 点击「操作日志」，可看到所有操作记录

### 失败链路验证

#### 1. 超出营业时段 → 不能创建
- 选「会议室B」（开放 09:00-18:00）
- 设开始 08:00 结束 10:00 → 提交，会报错：「申请时段超出场地营业时间」
- 或设开始 17:00 结束 19:00 → 同样报错

#### 2. 同场地同时段审批冲突 → 拦住后者
- 对同一场地同一时段提交 2 个申请（状态都是「待审批」）
- 审批通过第 1 个 → 成功
- 审批通过第 2 个 → 报错：「时段冲突：与申请 #X「活动名」时间重叠」

#### 3. 超出日配额 → 审批时拦截
- 选「活动室C」（日配额 2 场）
- 提交 3 个不重叠的申请
- 审批通过前 2 个 → 成功
- 审批第 3 个 → 报错：「超出当日配额（已确认 2 场，配额 2 场）」

#### 4. 取消后可撤销 → 恢复之前状态
- 找一个已确认的申请，点「取消」，填原因 → 状态变为「已取消」
- 点「撤销取消」→ 恢复为「已确认」（会重新校验冲突和配额）
- 查看详情的状态历史，可看到完整链路

#### 5. 撤销时冲突 → 不能撤销
- 申请 A：10:00-12:00，已确认 → 取消 → 状态为「已取消」
- 申请 B：10:00-12:00，审批通过 → 状态为「已确认」
- 尝试撤销申请 A 的取消 → 报错：时段冲突，无法恢复

### 数据持久化验证

1. 做一些操作（创建场地、提交申请、审批通过）
2. 记下已确认申请数量、审批意见、日志条数
3. 停止应用（Ctrl+C）
4. 重新启动：`python app.py`
5. 打开页面检查：
   - 场地信息、配额不变
   - 申请单状态、审批意见、取消原因都在
   - 状态历史链路完整
   - 操作日志保留
   - 排期数据对得上

## API 列表

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/venues` | 场地列表 |
| POST | `/api/venues` | 新增场地 |
| GET | `/api/venues/{id}` | 场地详情 |
| PUT | `/api/venues/{id}` | 更新场地 |

| GET | `/api/applications` | 申请列表（支持 venue_id/status/apply_date 过滤） |
| POST | `/api/applications` | 提交申请 |
| GET | `/api/applications/{id}` | 申请详情（含历史） |
| POST | `/api/applications/{id}/approve` | 审批通过 |
| POST | `/api/applications/{id}/reject` | 审批驳回 |
| POST | `/api/applications/{id}/cancel` | 取消申请 |
| POST | `/api/applications/{id}/revoke` | 撤销取消 |
| GET | `/api/schedule/{date}` | 当日排期视图 |
| GET | `/api/schedule/{date}/export` | 导出当日排期 CSV |
| GET | `/api/audit-logs` | 操作日志 |

## 状态机

```
已提交 (submitted)
  ↓ 系统自动流转
待审批 (pending_approval)
  ├─→ 已确认 (confirmed)   [审批通过]
  ├─→ 已驳回 (rejected)    [审批驳回]
  └─→ 已取消 (cancelled)   [取消申请]

已确认 (confirmed)
  └─→ 已取消 (cancelled)   [取消申请]

已取消 (cancelled)
  └─→ 之前的状态           [撤销取消，重校验]
```

每次状态变更都会写入 `status_history` 表，记录操作人、时间、原因。
