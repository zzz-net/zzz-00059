# 活动场地排期系统

本地运行的活动场地申请、审批、排期管理系统。核心围绕**申请-审批-冲突检测-历史链路**设计，不是简单的预约 CRUD。

## 功能特性

### 核心业务
- **场地管理**：维护场地信息、开放时段、日配额
- **申请流程**：提交 → 待审批 → 已确认 / 已驳回 / 已取消
- **冲突检测**：时段重叠校验、日配额校验
- **状态回退**：已取消申请可撤销回之前状态（会再次校验冲突）
- **历史链路**：每次状态变更都记录操作人、时间、原因，形成完整审计链
- **操作日志**：全局审计日志，记录所有关键操作
- **排期导出**：按日期导出 CSV 排期表
- **批量导入**：CSV 批量导入排期数据，支持预演、确认、取消和完整复核
- **角色分层**：审批人和普通申请人看到不同层级的字段，审批侧明细不会泄露给申请人

### 关键校验规则
1. **创建时**：时段必须在场地营业时间内
2. **审批时**：同场地同时段已确认的申请会拦截后者
3. **审批时**：当日已确认数量不能超过日配额
4. **撤销取消时**：重新检查时段冲突和日配额
5. **重复导入时**：同一 CSV 文件重复上传会被预演阶段检测到冲突
6. **已取消批次重复上传**：同一文件名 + 同一操作人对应的已取消批次再次上传会被拦截，判为历史重复，不能重新建单

### 权限规则
- **审批人**：可以审批通过/驳回、撤销取消、取消任意申请、上传/确认/取消导入批次、查看完整批次摘要和审批明细、导出批次复核 CSV
- **普通申请人**：只能提交申请、取消自己的申请，查看自己的排期结果和必要状态，**不能**看到批次 ID、导入文件名、审批人、审批结论、失败分类等审批侧明细
- 默认审批人名单：`张三`、`管理员`、`admin`、`Administrator`（可在 `app.py` 的 `APPROVERS` 中修改）

### 角色分层字段规则

| 接口 | 审批人可见 | 申请人可见 |
|------|-----------|-----------|
| 批次列表 `/api/import` | 批次ID、文件名、导入人、确认人、错误分类、审批状态聚合等全部字段 | 仅状态、成功/失败/总数、创建/更新时间 |
| 批次详情 `/api/import/{id}` | 全部字段 + 操作日志 + 关联申请日志 | 仅自己相关的记录，且不含 batch_id、error_category、conflict_with_application_id 等；application 子对象仅含 id/status/status_label |
| 排期视图 `/api/schedule/{date}` | 全部已确认申请字段 | 仅自己名下的已确认申请，且不含审批人、审批结论、冲突摘要等 |
| 排期导出 `/api/schedule/{date}/export` | 含审批人、审批意见、冲突摘要、审批结论、导入批次ID、导入文件名等18列 | 仅日期、场地、活动名称、申请人、开始时间、结束时间、参与人数、状态8列 |
| 申请列表 `/api/applications` | 全部字段 + 预检信息 | 不含审批人、审批结论、冲突摘要、审批意见、取消原因等 |
| 我的排期 `/api/my-schedule` | 全部字段（审批人查自己） | 不含审批人、审批结论等审批侧明细 |

## 本地启动

### 环境要求
- Python 3.8+

### 步骤

```bash
# 1. 进入项目目录
cd zzz-00059

# 2. 安装依赖
pip install -r requirements.txt

# 3. 启动应用（首次启动会自动建表并初始化3个示例场地）
python app.py
```

启动后访问：http://localhost:5001

数据存储在项目根目录下的 `scheduling.db`（SQLite 文件），重启不丢失。

## 验证步骤

### 主流程验证（合法申请从提交到审批通过 → 导出）

1. **启动应用**：`python app.py`
2. **查看场地**：打开首页，确认有 3 个示例场地（多功能厅A、会议室B、活动室C）
3. **新建申请**：
   - 点击「申请管理」→「+ 新建申请」
   - 选择场地「多功能厅A」，填活动名称、申请人
   - 选明天日期，开始 10:00，结束 12:00
   - 提交，状态为「待审批」
4. **审批通过**：
   - 点击「审批面板」，找到刚提交的申请
   - 点「通过」，填写审批意见
   - 状态变为「已确认」
5. **查看详情与历史**：
   - 点「详情」，可看到完整的状态历史链路：提交 → 待审批（系统流转）→ 已确认（审批人 + 意见）
6. **查看排期**：
   - 点击「排期视图」，选择对应日期
   - 可看到该场地的已确认排期
7. **导出排期**：
   - 点「导出CSV」，下载当日排期表
   - 用 Excel 打开，确认数据正确
8. **查看操作日志**：
   - 点击「操作日志」，可看到所有操作记录

### 批量导入验证

1. **上传 CSV**：在「批量导入」区域选择 CSV 文件，填写操作人（需为审批人），点击上传
2. **查看预演结果**：上传后系统自动预演，显示通过/失败条数和失败原因分类
3. **确认导入**：点击「确认导入」，系统正式写入申请数据
4. **查看批次详情**：点击「查看详情」，可看到每条记录的导入状态、关联申请、错误分类
5. **导出批次复核 CSV**：点击「导出CSV」，下载含完整审批状态和失败分类的复核表

#### 已取消批次重复上传拦截

1. 上传一个 CSV 文件，在预演阶段点击「取消」
2. 再次上传同一文件名（同一操作人）→ 返回 409 错误：「属于历史重复，不能再重新建单」
3. 换一个不同的文件名上传 → 不受影响，正常创建批次

### 角色分层验证

1. **申请人查看批次列表**：用非审批人身份调用 `/api/import?operator=李四`，返回的批次不含 ID、文件名、错误分类等审批侧字段
2. **申请人查看批次详情**：用非审批人身份查看自己有关的批次详情，记录中不含 batch_id、error_category、conflict_with_application_id
3. **申请人排期导出**：导出 CSV 仅含 8 列基本字段，不含审批人、审批结论等
4. **审批人完整复核**：用审批人身份可查看全部字段，包括操作日志、失败分组、审批状态聚合

### 失败链路验证

#### 1. 超出营业时段 → 不能创建
- 选「会议室B」（开放 09:00-18:00）
- 设开始 08:00 结束 10:00 → 提交，会报错：「申请时段超出场地营业时间」
- 或设开始 17:00 结束 19:00 → 同样报错

#### 2. 同场地同时段审批冲突 → 拦住后者
- 对同一场地同一时段提交 2 个申请（状态都是「待审批」）
- 审批通过第 1 个 → 成功
- 审批通过第 2 个 → 报错：「时段冲突：与申请 #X「活动名」时间重叠」

#### 3. 超出日配额 → 审批时拦截
- 选「活动室C」（日配额 2 场）
- 提交 3 个不重叠的申请
- 审批通过前 2 个 → 成功
- 审批第 3 个 → 报错：「超出当日配额（已确认 2 场，配额 2 场）」

#### 4. 取消后可撤销 → 恢复之前状态
- 找一个已确认的申请，点「取消」，填原因 → 状态变为「已取消」
- 点「撤销取消」→ 恢复为「已确认」（会重新校验冲突和配额）
- 查看详情的状态历史，可看到完整链路

#### 5. 撤销时冲突 → 不能撤销
- 申请 A：10:00-12:00，已确认 → 取消 → 状态为「已取消」
- 申请 B：10:00-12:00，审批通过 → 状态为「已确认」
- 尝试撤销申请 A 的取消 → 报错：时段冲突，无法恢复

### 数据持久化验证

1. 做一些操作（创建场地、提交申请、审批通过）
2. 记下已确认申请数量、审批意见、日志条数
3. 停止应用（Ctrl+C）
4. 重新启动：`python app.py`
5. 打开页面检查：
   - 场地信息、配额不变
   - 申请单状态、审批意见、取消原因都在
   - 状态历史链路完整
   - 操作日志保留
   - 排期数据对得上
   - 批次列表和详情与重启前一致
   - 角色分层的字段过滤与重启前一致
   - 排期导出 CSV 内容与重启前一致

## 状态机

```
已提交 (submitted)
  ↓ 系统自动流转
待审批 (pending_approval)
  ├─→ 已确认 (confirmed)   [审批通过]
  ├─→ 已驳回 (rejected)    [审批驳回]
  └─→ 已取消 (cancelled)   [取消申请]

已确认 (confirmed)
  └─→ 已取消 (cancelled)   [取消申请]

已取消 (cancelled)
  └─→ 之前的状态           [撤销取消，重校验]
```

### 导入批次状态

```
预演中 (preview)
  ├─→ 已确认待导入 (confirmed)  [审批人确认]
  │     └─→ 已完成 (completed)  [正式导入执行]
  └─→ 已取消 (cancelled)        [审批人取消]

已取消 (cancelled)
  └─→ 不可重新上传同文件名      [历史重复拦截]
```

每次状态变更都会写入 `status_history` 表，记录操作人、时间、原因。
